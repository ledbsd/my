Фикстуры в pytest Как правило, любые тесты предполагают использование данных, подготовленных специально для тестирования: чтобы произвести какие-то манипуляции с объектом, этот объект сначала надо создать. При работе с библиотекой unittest данные для тестирования создаются в фикстурах setUp() и setUpClass(). В pytest тоже применяются фикстуры: это функции, обозначенные маркером @pytest.fixture. В unittest для каждого класса нужно создавать собственные фикстуры, иногда дублируя их: например, в пройденных уроках пришлось в нескольких местах одинаковыми фикстурами создавать пользователей. В отличие от unittest, в библиотеке pytest фикстура создаётся только один раз, а затем вызывается во всех нужных тестах. Вот некоторые особенности фикстур в pytest: Каждая фикстура называется уникальным именем и вызывается явно, как параметр в тестирующей функции. Вызывать фикстуры можно не только в параметрах теста, но и через маркеры. Фикстуры реализованы по модульному принципу: первая фикстура может вызывать вторую, вторая — третью и так далее. Так можно набрать целые цепочки из фикстур, подготавливающих необходимое окружение. Фикстурам могут быть назначены области видимости (scope), чтобы они вызывались не для каждой тестовой функции или метода, а, например, единожды для класса, модуля или всей тестовой сессии целиком. Подготовка к тестированию и очистка после тестов (по-английски — setup и teardown) пишутся в одной функции-фикстуре; это делает код более читаемым: всё необходимое хранится в одной функции. Фикстуры можно хранить отдельно от тестов, в специальном файле conftest.py; pytest самостоятельно обнаруживает фикстуры в этом файле, и они становятся доступными во всех тестах. Импортировать фикстуры в каждый файл с тестами нет необходимости. Как это делается в pytest Если в тестирующей функции необходимо применить объекты из фикстуры — название фикстуры передают в параметры функции. При запуске тестов pytest находит нужную фикстуру по имени, указанному в параметрах тестирующей функции — и передаёт в функцию объект, который возвращает фикстура. import pytest @pytest.fixture # Декоратор, обозначающий, что эта функция - фикстура. def give_me_a_string(): return 'Какой чудесный день!' # Фикстура возвращает строку. # Если тестовой функции для работы нужна фикстура, # она указывается в параметрах. def test_string_fixture(give_me_a_string): # Переменная с именем фикстуры содержит в себе объект, # который вернула фикстура. assert give_me_a_string[0] == 'К' Здесь фикстура возвращает строку, а тест проверяет, что строка начинается с буквы К. Добавим еще одну фикстуру, которая будет вызывать уже существующую фикстуру give_me_a_string() и «упаковывать» строку в список. import pytest @pytest.fixture def give_me_a_string(): return 'Какой чудесный день!' # Новая фикстура возвращает список со строкой из первой фикстуры. @pytest.fixture def pack_to_list(give_me_a_string): # Фикстура может вызывать другие фикстуры. return [give_me_a_string] # Тестовая функция использует обе фикстуры и проверяет их содержимое. def test_string_fixture(pack_to_list, give_me_a_string): assert pack_to_list == [give_me_a_string] Испортим тест: вместо проверки на равенство == поставим проверку на неравенство: !=. ... def test_string_fixture(pack_to_list, give_me_a_string): assert pack_to_list != [give_me_a_string] При выполнении теста проверка провалится, и в отчёт будут выведены значения фикстур, на которых упал тест. ======================================= FAILURES ======================================= _________________________________ test_string_fixture __________________________________ pack_to_list = ['Какой чудесный день!'], give_me_a_string = 'Какой чудесный день!' def test_string_fixture(pack_to_list, give_me_a_string): > assert pack_to_list != [give_me_a_string] E AssertionError: assert ['Какой чудесный день!'] != ['Какой чудесный день!'] Вызов фикстуры при помощи маркера Иногда фикстура должна просто создать какой-то объект или запустить процесс, но при этом она не возвращает никаких значений или же возвращаемые значения не используются в тестирующей функции. В проекте pytest_trial создайте два файла: engine_class.py и test_engine.py; скопируйте в них код из листингов: # engine_class.py class Engine: """Класс двигателя.""" def __init__(self): # При создании объекта двигателя он не запущен. self.is_running = False Из файла engine_class.py мы будем только импортировать класс двигателя и ничего больше. Теперь тесты: # test_engine.py import pytest # Импортируем класс двигателя. from engine_class import Engine @pytest.fixture def engine(): """Фикстура возвращает экземпляр класса двигателя.""" return Engine() # Эта фикстура не возвращает никаких значений, но изменяет объект, # созданный другой фикстурой. @pytest.fixture def start_engine(engine): # Вызываем фикстуру получения объекта двигателя. """Фикстура запускает двигатель.""" # Изменяем значение свойства объекта: engine.is_running = True def test_engine_is_running(engine, start_engine): # Вызываем обе фикстуры. """Тест проверяет, работает ли двигатель.""" assert engine.is_running # Проверяем, что значение атрибута is_running это True. Порядок указания фикстур в параметрах на результат не влияет: pytest «под капотом» сам выстраивает их в нужном порядке. Запустите тест. Фикстура start_engine указана в параметрах теста test_engine_is_running(); без фикстуры start_engine тест провалится. Но эта фикстура не возвращает никаких объектов и не используется в самом тесте; получается, что параметр start_engine не применяется в функции. В таких случаях фикстуру можно вызывать не через параметр, а через маркер @pytest.mark.usefixtures. Для этого в параметре декоратора нужно указать название фикстуры в виде строки: # Добавляем маркер и указываем название фикстуры строкой. @pytest.mark.usefixtures('start_engine') def test_engine_is_running(engine): # А из параметров функции фикстуру start_engine убираем. assert engine.is_running Если фикстур нужно несколько — их названия можно перечислять через запятую: @pytest.mark.usefixtures('one_fixture', 'another_fixture') Обратите внимание, что такой способ работает только с тестирующими функциями. Если в одной фикстуре нужно вызвать другую — этот маркер не сработает! Если определённую фикстуру нужно применить ко всем тестам класса, то применяется тот же декоратор @pytest.mark.usefixtures, только ставится он перед названием класса. А для применения фикстуры ко всем тестам модуля, как и в случае с обычным маркером, нужно использовать зарезервированную глобальную переменную: pytestmark = pytest.mark.usefixtures('some_fixture'). Автоматически используемые фикстуры В pytest можно настроить фикстуру так, чтобы она автоматически применялась ко всем тестам. Для этого при объявлении фикстуры в декораторе @pytest.fixture указывают параметр autouse=True. Такие фикстуры могут использоваться для действий, которые требуются для всех тестов сразу. Продолжим пример с двигателем: объявим start_engine() автоматически вызываемой фикстурой. Добавьте в декоратор фикстуры start_engine параметр autouse=True и уберите start_engine из параметров теста test_engine_is_running(): # test_engine.py import pytest from engine_class import Engine @pytest.fixture def engine(): """Фикстура возвращает экземпляр класса двигателя.""" return Engine() @pytest.fixture(autouse=True) # Обозначаем фикстуру как автоматически вызываемую. def start_engine(engine): """Фикстура запускает двигатель.""" engine.is_running = True # Вызываем только одну фикстуру. # Запуск двигателя выполнится автоматически, без вызова. def test_engine_is_running(engine): """Тест проверяет, работает ли двигатель.""" assert engine.is_running Скопируйте код из листинга в файл test_engine.py и запустите тесты: pytest test_engine.py -v Несмотря на то, что в параметрах теста фикстура start_engine() не вызывается — тест пройден и вернул OK: двигатель запущен. Для эксперимента уберите из декоратора параметр autouse=True и вновь запустите тест. Тест ожидаемо провалился. Вторая фикстура на месте, но она не вызывается: в параметрах теста она не вызвана, а без autouse=True она сама не выполнится. Области видимости фикстуры Некоторые операции при тестировании могут быть довольно затратными по ресурсам и длительными по времени: например — создание набора файлов или создание и наполнение базы данных. В то же время для некоторых тестов может быть достаточно, чтобы фикстура была выполнена только один раз для класса, модуля или тестовой сессии (при выполнении всех найденных тестов). За то, как выполняются фикстуры, отвечают области видимости фикстур (scope). Любая фикстура выполняется при вызове; результат выполнения фикстуры кешируется, созданные объекты сохраняются в памяти; а вот удаление хранящихся объектов происходит в зависимости от области видимости. В pytest есть такие области видимости: Область видимости функции function (по умолчанию): созданные в фикстурах объекты удаляются после выполнения теста. Область видимости класса class: созданные в фикстурах объекты удаляются после выполнения последнего теста класса. Область видимости модуля module: созданные в фикстурах объекты удаляются после выполнения последнего теста модуля. Область видимости пакета package: созданные в фикстурах объекты удаляются после выполнения последнего теста пакета (директории с тестами). Область видимости сессии (session): созданные в фикстурах объекты удаляются в конце тестовой сессии (после выполнения всех найденных тестов). Чтобы понять, как это работает, продолжим разбирать пример с двигателями. Добавим еще один тест, проверяющий, что объект, возвращаемый фикстурой engine — это экземпляр класса Engine. Чтобы отследить, сколько раз была вызвана фикстура engine, добавим в неё print(). import pytest from engine_class import Engine @pytest.fixture def engine(): """Фикстура возвращает экземпляр класса двигателя.""" print('Engine factory') # Добавьте вывод сообщения. return Engine() @pytest.fixture(autouse=True) def start_engine(engine): """Фикстура запускает двигатель.""" engine.is_running = True def test_engine_is_running(engine): """Тест проверяет, работает ли двигатель.""" assert engine.is_running # Допишите новый тест. def test_check_engine_class(engine): """Тест проверяет класс объекта.""" assert isinstance(engine, Engine) Запустите тесты с ключами -s , чтобы вывод print() был виден в терминале, и -qq, чтобы отсечь лишнюю информацию. В отчёте будет такой вывод: Engine factory .Engine factory. Прошло два теста и для каждого вызывалась фикстура engine. Но при выполнении этих двух тестов не важно, тестируется ли один и тот же экземпляр двигателя или два разных. Лучше тестировать один: это позволит сэкономить ресурсы. Для этого нужно настроить фикстуру так, чтобы она выполнялась один раз для всех тестов (а не для каждого теста в отдельности). Укажем для фикстуры engine область видимости 'session'; при такой области видимости результат однократного выполнения фикстуры будет храниться для всех вызовов. Добавьте к декоратору фикстуры engine параметр scope со значением 'session': @pytest.fixture(scope='session') def engine(): ... И запустите тест еще раз: pytest -s -qq На этот раз отчёт будет таким: Engine factory .. Было выполнено два теста, каждый из них вызывал фикстуру engine, но сама фикстура была выполнена только один раз: её результат был закеширован для всех вызовов. Области видимости фикстур применяют в тех случаях, когда нет необходимости вызывать одну и ту же фикстуру для каждого теста отдельно, а можно вызвать её для всех тестов сразу (scope='session'), для пакета с тестами (scope='package'), модуля (scope='module') или класса (scope='class'). Очистка (teardown) После тестов нужно прибрать за собой. Некоторые создаваемые объекты в конце тестов должны быть удалены явным образом, например — соединение с БД. В случае, если перед выполнением тестов создавался набор тестовых файлов — по окончании тестов эти файлы нужно удалить. Код фикстуры должен быть разделён: часть кода выполняется до начала тестов (setup), а часть — по их окончании (teardown). Схематически порядок работы будет такой: Фикстура (setup) => Тестирующие функции => Фикстура (teardown) Чтобы в коде фикстуры обозначить момент, когда надо переходить к тестам, используется ключевое слово yield; если фикстура возвращает какой-то объект, то yield используется вместо return. После того, как тесты будут выполнены, продолжит выполняться код, описанный в фикстуре после строки с yield. Проиллюстрируем это на примере с двигателями. Будем заводить двигатель перед каждым тестом и глушить его после каждого теста. Дополним фикстуру запуска двигателя соответствующими операциями, а для контроля расставим print() по всему коду. import pytest from engine_class import Engine @pytest.fixture(scope='session') def engine(): """Фикстура возвращает экземпляр класса двигателя.""" print('Engine factory') return Engine() @pytest.fixture(autouse=True) def start_engine(engine): """Фикстура запускает двигатель.""" engine.is_running = True # Запустим двигатель. # Распечатаем строчку до выполнения теста. print(f'Before test engine.is_running {engine.is_running}') yield # В этот момент начинает выполняться тест. engine.is_running = False # Заглушим двигатель. # Распечатаем строчку после выполнения теста и остановки двигателя. print(f'After test engine.is_running {engine.is_running}') def test_engine_is_running(engine): """Тест проверяет, работает ли двигатель.""" print('test_engine_is_running') # Выведем название теста. assert engine.is_running def test_check_engine_class(engine): """Тест проверяет класс объекта.""" print('test_check_engine_class') # Выведем название теста. assert isinstance(engine, Engine) Запустите тесты pytest -s -qq. В отчёте должны отобразиться такие строки: Engine factory Before test engine.is_running True test_engine_is_running .After test engine.is_running False Before test engine.is_running True test_check_engine_class .After test engine.is_running False Выполнены два теста, в отчёте их выполнение обозначено точками. Видно, что строки печатаются именно так, как задумано: после каждого теста управление переходит обратно к фикстуре, которая глушит двигатель. Таким образом запуск нужного процесса и «уборка за собой» происходит в одной функции-фикстуре. Аналогичным образом можно создавать и закрывать соединения с БД, создавать и удалять файлы или любые объекты. conftest.py Фикстуры в pytest можно хранить в специальном файле с зарезервированным названием conftest.py. В зависимости от того, какие фикстуры в нём сохранены, этот файл можно хранить по-разному: в корне проекта — если фикстуры или настройки относятся ко всему проекту; в директориях с тестами конкретных Django-приложений — если фикстуры относятся к одному конкретному приложению. Если создать несколько разных файлов conftest.py, то файлы, сохранённые в Django-приложениях, будут расширять набор фикстур в корневом conftest.py или переопределять размещённые в нём одноимённые фикстуры, если такие есть. Фикстуры, сохранённые в conftest.py, не требуется импортировать в файлы с тестирующими функциями. Создайте файл conftest.py; перенесите из файла test_engine.py две фикстуры и импорт класса двигателя. Заодно уберите из кода функции print() — они уже выполнили своё назначение. # conftest.py import pytest from engine_class import Engine @pytest.fixture(scope='session') def engine(): """Фикстура возвращает экземпляр класса двигателя.""" return Engine() @pytest.fixture(autouse=True) def start_engine(engine): """Фикстура запускает двигатель.""" engine.is_running = True yield engine.is_running = False В файле test_engine.py должны остаться только тестирующие функции. Импортировать фикстуры в этот файл не нужно: импортируем только класс двигателя. # test_engine.py from engine_class import Engine def test_engine_is_running(engine): """Тест проверяет, работает ли двигатель.""" assert engine.is_running def test_check_engine_class(engine): """Тест проверяет класс объекта.""" assert isinstance(engine, Engine) Теперь запустите тесты для файла test_engine.py. Всё работает! Фикстуры хранятся в отдельном файле, импорта нет, но pytest и без посторонней помощи находит нужные фикстуры и запускает их.
